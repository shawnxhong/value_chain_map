# VCRG 设计文档修改意见与依据

**面向**：与 GPT 讨论用  
**对应原文**：`value_chain_research_tool_prd.md` (v0.1)  
**修改后文档**：`value_chain_map_design_v0.2.md`  
**日期**：2026-06-30

---

## 0. 一句话总纲

> 把这个项目从"一个什么都想做的产业链投资工具"，**收窄重定位为科技股五层分析框架中第二层（行业结构与价值链）的专用工具**。它的产出不是买卖结论，而是一张证据化的产业链地图 + 一份带"结构性论点"的公司分层名单，然后**交棒**给第三/四/五层的工具。

原文 v0.1 工程架构做得很满，但在"投资价值如何产生"上偏虚，且边界不清——它同时想做地图、瓶颈、利润池、排雷、报告生成，结果每一项都不够深。重定位到第二层后，成功标准、数据源、schema、打分方式都应随之改写。

## 1. 修改意见总表（按重要性排序）

| # | 修改项 | 原文位置 | 修改方向 | 修改依据 |
|---|---|---|---|---|
| M1 | **重定位为"第二层地图引擎"，而非投资决策工具** | §1、§3 | 明确"产出地图与结构名单，不产出买卖结论"；成功标准从"理解速度/排雷"改为"能说清谁强谁弱、利润在谁手里、利润往哪移，且句句有出处" | 项目被明确限定为五层框架的第二层；"收窄到个股决策"是整个工具矩阵的事，不是单个工具的事 |
| M2 | **建立与其他四层的输入/输出接口契约** | 全文缺失 | 新增"层间接口"章节：输入=第一层的需求潮水/利率状态；输出=每家公司的"结构画像卡"，移交三/四/五层 | 既然是工具矩阵，接口比功能更重要；输出契约是这个工具最该打磨的东西 |
| M3 | **把"科技独有结构性力量"建成一等公民** | §7、§10 几乎缺失 | 技术路线迁移/S曲线、capex 周期、赢家通吃/集中度、扩产周期——作为核心建模对象，而非静态供应商网络 | 框架第二层明确要求理解这些动态力量；利润池会"移动"正是因为它们；静态供应商图是 FactSet/Bloomberg 已有的低差异化产物 |
| M4 | **财务数据角色重定义：只做"跨环节横向对比"，不做公司基本面** | §10.2、上次评审 | 用毛利率/营业利润率/ROIC 横向比，做利润池热力图；DCF/NRR/SBC/单位经济全部划归第四/五层 | "利润沉淀在谁手里"是第二层核心问题，需要财务数据，但只作结构性比较；公司内部分析越界 |
| M5 | **打分改为"结构位置分"且证据驱动的 checklist** | §10.2.1–10.2.4 | 只保留 Bottleneck / Profit Pool / Weak Link 三个纯结构分；取消加权小数公式，改为"命中几项特征 + 原文" | 原 Value Chain Strength Score 混入增长/ROIC 等第四层因子；加权小数是项目自己反对的"伪精确"(§20) |
| M6 | **数据源优先级倒置** | §14.3 P0/P1 | transcript + investor deck 升为 P0；10-K 降为"identity + 集中度占比"用途 | 原文自承 transcript 对投资最有用、10-K 供应链信息不完整；按原优先级会产出"节点多、边稀、低信息量"的图 |
| M7 | **置信度去公式化** | §15.2 | 取消 5 项加权公式；改为离散 high/medium/low，由"来源类型 + 表述直接性"两维决定 + 一句话理由 | 与项目自身风险条目"置信度伪精确"(§20)自相矛盾；无标注数据时连续分数无可解释含义 |
| M8 | **schema 大幅收缩** | §7、§18 | P0 节点：Company/Product/EndMarket/Technology/Document+Evidence；P0 边：SUPPLIES_TO/COMPETES_WITH/SERVES_MARKET/PRODUCES/MIGRATES_TO；其余推后 | 11 节点 13 边对 MVP 过重；投资价值来自深度非节点数量(§21 决策3) |
| M9 | **技术选型降配** | §13 | MVP 用 Kuzu 或 Postgres+NetworkX；Neo4j 推迟到协作版 | 50–500 条边规模下 Cypher 价值未体现，Neo4j 是多一个要运维的服务；匹配"个人研究者"画像 |
| M10 | **大幅减少 autonomous agent** | §9 | P0 用确定性函数 + 2 个 prompt（抽取 + 校验）+ 人工 review；10 个 agent 推后 | 原文自问应否更状态机化(§22 Q5)；MVP 不需要 Planner/Discovery/Diff 等 7 个 agent |
| M11 | **匿名大客户解析升为核心工作流** | §11.2 | "匿名大客户节点 + 后续身份解析"作为一等公民 | "某客户占23%"是 10-K 最可靠的产业链信号，是相对纯 LLM 问答的真正优势 |
| M12 | **时间/失效（temporal）强化** | §7.3 弱、§19 边缘 | 每条边强制 `as_of_date`；UI 默认提示"该关系最后验证于 N 个月前" | 原文最大痛点之一是"关系变化难维护"(§2.4)，但 MVP 把 temporal 边缘化 |
| M13 | **MVP 收窄到一条子链打透** | §14 全行业类别 | 从 HBM → 先进封装 → GPU → 超大厂 这一条链做透，而非十几个类别铺开 | 先做深再做广(§5.4)；这条链技术迁移和瓶颈都典型，最能展示动态建模价值 |
| M14 | **新增"结构画像卡"作为核心交付物与评估单元** | 全文缺失 | 每家公司输出结构化画像卡（结构位置/利润池层级/关键依赖/技术迁移风险/结构性论点/移交建议/证据/分层结论） | 它是工具矩阵的咬合点，把"地图"收敛成"可移交的下一步动作" |
| M15 | **新增早期评估黄金集** | §22 Q15 | 建 20–30 条人工标注的"黄金关系集"，每次改 prompt 跑 precision/recall | 没有它无法判断抽取在变好还是变坏 |

## 2. 予以保留的优秀设计（不改）

- **四层 fact / estimate / inference / thesis（§5.2）**——是整份设计的灵魂，务必保留并强化 UI 视觉区分。
- **production / staging 分离 + LLM 不直接写主图（§5.3）**——正确。
- **证据作为独立对象（§7.4）**——正确，但 P0 字段精简。
- **"图谱不是事实本身，证据才是"（§5.1）**——正确。
- **§20 风险清单、§21 关键决策**——大体保留，仅按重定位微调。

## 3. 重定位带来的边界（IN / OUT）

| 维度 | 本工具负责（IN，第二层） | 交给别的层（OUT） |
|---|---|---|
| 价值链结构 | 谁供给谁、谁竞争、谁替代、谁是瓶颈/弱势 | —— |
| 利润池 | 跨环节横向比毛利/ROIC，定位利润沉淀点 | 单公司估值（第五层） |
| 护城河 | 给出**环节级结构性假设**（高切换成本/赢家通吃倾向） | 公司级护城河深度与可被颠覆性判断（第三层） |
| 基本面 | 仅用聚合/对比财务指标 | 三张表、NRR、SBC、单位经济（第四层） |
| 宏观 | 不分析，仅作为输入接进来 | 利率/流动性判断（第一层） |
| 最终动作 | 产出分层候选名单 | 个股买卖决策（五层收窄后的整体动作） |

**判定规则**：用财务数据时，问自己是在比较"环节之间"还是分析"一家公司内部"——前者属第二层（保留），后者交出去。

## 4. 建议你向 GPT 提问的重点

1. 重定位为"第二层地图引擎"是否认同？若不认同，工具的成功标准应如何定义？
2. "结构画像卡"作为核心交付物与层间接口，字段是否完备？
3. 把技术迁移/S曲线/capex周期/集中度建成一等公民，是否同意优先于静态供应商网络？
4. 数据源优先级倒置（transcript/deck 为 P0）在工程可行性上是否有阻碍（transcript 获取成本）？
5. schema 收缩到 5 节点 5 边是否过激？哪些必须更早加回？
6. 打分去公式化、改证据 checklist 是否会牺牲可比较性？如何在不伪精确的前提下支持排序？
